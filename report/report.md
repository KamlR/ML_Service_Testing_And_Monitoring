## Надёжность и отказоустойчивость

Ниже будут описаны произведённые доработки кода сервиса для повышения его надёжности и отказоустойчивости.

### Описание проблемы №1
Сервис не обрабатывал ситуацию, когда в `/predict` приходят не все фичи, необходимые для текущей модели. Из-за этого в пайплайн могли передаваться `None`, что приводило к ошибкам уже на этапе инференса.

### Решение проблемы №1
В функции `to_dataframe` добавлена явная проверка обязательных для активной модели признаков. Если хотя бы одна нужная фича отсутствует, сервис возвращает контролируемую ошибку `422` с перечислением недостающих полей.

---

### Описание проблемы №2
Сервис не обрабатывал ситуацию, когда в ручку `/updateModel` передавался невалидный `run_id` — пустой, состоящий из пробелов или несуществующий в MLflow.

### Решение проблемы №2
В `Model.set()` добавлена валидация `run_id` и обработка ошибок загрузки модели из MLflow. При неуспешной загрузке выбрасывается `ModelLoadError`, который в ручке `/updateModel` преобразуется в HTTP-ответ `400`. При этом старая модель в сервисе не затирается и продолжает использоваться.

---

### Описание проблемы №3
Если модель не загружалась при старте приложения, сервис не был устойчив к такой ситуации: приложение могло завершиться с ошибкой и не подняться вовсе.

### Решение проблемы №3
В `lifespan` загрузка стартовой модели обёрнута в `try/except`. Если загрузка не удалась, сервис всё равно стартует, а состояние отражается в `/health` как `degraded`. Это позволяет сохранить доступность сервиса и упростить диагностику проблемы.

---

### Описание проблемы №4
В `/predict` не было явной обработки ошибок предобработки и ошибок инференса модели, из-за чего пользователю могли возвращаться неочевидные внутренние ошибки.

### Решение проблемы №4
В обработчик `/predict` добавлена раздельная обработка:
- ошибок валидации и предобработки входных данных — ответ `422`;
- ошибок выполнения модели на этапе инференса — ответ `500` с контролируемым сообщением `Model inference failed`.

---

### Описание проблемы №5
Сервис не учитывал ситуацию, когда загруженная модель не содержит `feature_names_in_`, а логика выбора нужных признаков целиком на это опирается.

### Решение проблемы №5
В свойстве `MODEL.features` добавлена проверка наличия `feature_names_in_`. Если модель загружена в некорректном формате и не содержит список фичей, выбрасывается понятная ошибка `ModelLoadError`.




## Тесты для сервиса

### Тест1 (Предобработка: выбор нужных фич)
Проверяет, что функция `to_dataframe` корректно оставляет только те признаки, которые нужны текущей модели. Это важно, потому что разные модели в MLflow могут быть обучены на разных наборах фич.

### Тест2 (Предобработка: отсутствие обязательной фичи)
Проверяет, что при отсутствии обязательного признака функция предобработки выбрасывает контролируемую ошибку. Это нужно, чтобы сервис не падал глубже в пайплайне и возвращал понятную ошибку пользователю.

### Тест3 (Предобработка: неподдерживаемая фича)
Проверяет, что функция не пропускает неизвестные или неподдерживаемые признаки в список `needed_columns`. Это защищает сервис от некорректной конфигурации модели.

### Тест4 (Загрузка модели: успешная активация)
Проверяет, что контейнер `Model` успешно загружает модель, сохраняет `run_id` и делает доступным список признаков через `feature_names_in_`.

### Тест5 (Загрузка модели: невалидный run_id)
Проверяет, что при пустом или некорректном `run_id` выбрасывается контролируемая ошибка `ModelLoadError`, а не происходит необработанное падение.

### Тест6 (Инференс модели)
Проверяет, что загруженная модель умеет возвращать вероятности в ожидаемом формате. Это базовая проверка совместимости модели с сервисом.

### Тест7 (Хэндлер `/predict`: успешный запрос)
Проверяет, что ручка `/predict` при валидном запросе и загруженной модели возвращает HTTP 200 и корректные поля `prediction` и `probability`.

### Тест8 (Хэндлер `/predict`: отсутствует обязательная фича)
Проверяет, что ручка `/predict` возвращает HTTP 422, если для текущей модели не передан обязательный признак.

### Тест9 (Хэндлер `/predict`: модель не загружена)
Проверяет, что при отсутствии активной модели сервис корректно возвращает HTTP 503.

### Тест10 (Валидация запроса `/predict`)
Проверяет, что FastAPI/Pydantic отклоняет невалидный JSON-запрос, например если поле `age` передано строкой вместо числа.

### Тест11 (Хэндлер `/updateModel`: невалидный run_id)
Проверяет, что ручка обновления модели возвращает HTTP 400 при ошибке загрузки модели по неверному `run_id`.

### Тест12 (Хэндлер `/health`: degraded state)
Проверяет, что ручка `/health` показывает деградированное состояние сервиса, если модель не была загружена.


## Логируемые метрики
1. ml_service_http_requests_total - общее количество HTTP-запросов к сервису с разбиением по методу, пути и статус-коду ответа.

2. ml_service_http_request_duration_seconds - длительность обработки HTTP-запросов в секундах с разбиением по методу и пути. По этой гистограмме можно строить перцентили 75%, 90%, 95%, 99% и 99.9% через `histogram_quantile`.

3. ml_service_http_request_exceptions_total - количество необработанных исключений при обработке HTTP-запросов с разбиением по методу, пути и типу исключения.

4. ml_service_uptime_seconds - аптайм сервиса в секундах с момента запуска.

5. ml_service_process_cpu_percent - текущее потребление CPU процессом сервиса в процентах.

6. ml_service_process_memory_rss_bytes - текущее потребление оперативной памяти процессом сервиса в байтах.

7. ml_service_process_open_fds - текущее количество открытых файловых дескрипторов процессом сервиса.

8. ml_service_predict_requests_total - общее количество запросов в ручку `/predict`.

9. ml_service_predict_errors_total - количество ошибок в ручке `/predict` с разбиением по типу ошибки (`model_not_loaded`, `preprocessing_error`, `inference_error`).

10. ml_service_preprocessing_duration_seconds - время предобработки входных данных в секундах. По этой гистограмме можно строить перцентили 75%, 90%, 95%, 99% и 99.9%.

11. ml_service_model_inference_duration_seconds - время инференса модели в секундах. По этой гистограмме можно строить перцентили 75%, 90%, 95%, 99% и 99.9%.

12. ml_service_model_prediction_probability - распределение вероятностей положительного класса, возвращаемых моделью.

13. ml_service_model_predictions_total - количество предсказаний модели с разбиением по итоговому классу (`0` или `1`).

14. ml_service_feature_missing_total - количество случаев, когда обязательная для текущей модели фича отсутствовала во входном запросе, с разбиением по названию фичи.

15. ml_service_feature_numeric_value - распределение значений числовых признаков, поступающих в сервис, с разбиением по названию признака.

16. ml_service_feature_categorical_value_total - количество наблюдений категориальных значений признаков с разбиением по названию признака и конкретному значению.

17. ml_service_model_updates_total - общее количество успешных обновлений модели через ручку `/updateModel`.

20. ml_service_model_update_errors_total - общее количество неуспешных попыток обновления модели.

21. ml_service_model_last_update_timestamp_seconds - timestamp последнего успешного обновления модели.

22. ml_service_active_model_info - информация о текущей активной модели в проде, содержит `run_id` и тип модели (`LogisticRegression`, `RandomForestClassifier`, `GradientBoostingClassifier` и т.д.).

23. ml_service_active_model_feature_info - список признаков, необходимых для текущей активной модели, с разбиением по `run_id` и названию фичи.


## Grafana

### Блок1. Технические метрики сервиса.
1. RPS сервиса (HTTP requests per second)
```
sum(rate(ml_service_http_requests_total[1m]))
```

2. Доля 5xx ошибок (5xx error rate)
```
sum(rate(ml_service_http_requests_total{status_code=~"5.."}[5m]))
/
sum(rate(ml_service_http_requests_total[5m]))
```

3. Время ответа p95 (HTTP latency p95)
```
histogram_quantile(
  0.95,
  sum by (le) (rate(ml_service_http_request_duration_seconds_bucket[5m]))
)
```

### Блок2. Метрики, связанные с данными.

1. Время предобработки p95 (Preprocessing latency p95)
```
histogram_quantile(
  0.95,
  sum by (le) (rate(ml_service_preprocessing_duration_seconds_bucket[5m]))
)
```

2. Отсутствующие обязательные фичи (Missing required features)
```
sum by (feature) (increase(ml_service_feature_missing_total[1h]))
```

3. Распределение числовой фичи age (Age distribution)
```
sum by (le) (rate(ml_service_feature_numeric_value_bucket{feature="age"}[5m]))
```

### Блок3. Метрики работы модели.

1. Время инференса p95 (Inference latency p95)
```
histogram_quantile(
  0.95,
  sum by (le) (rate(ml_service_model_inference_duration_seconds_bucket[5m]))
)
```

2. Распределение вероятностей модели (Prediction probability distribution)
```
sum by (le) (rate(ml_service_model_prediction_probability_bucket[5m]))
```

3. Частота классов 0/1 (Predicted classes)
```
sum by (prediction) (increase(ml_service_model_predictions_total[1h]))
```

### Блок4. Информация по модели и её обновлению

1. Активная модель (Active model info)
ml_service_active_model_info

2. Набор активных фичей модели (Active model features)
ml_service_active_model_feature_info

3. Время последнего обновления модели (Last model update timestamp)
ml_service_model_last_update_timestamp_seconds

## Evidently

Мониторинг дрифта реализован с помощью библиотеки Evidently.

После каждого успешного запроса в `/predict` сервис сохраняет:
- признаки, которые использовала активная модель,
- предсказанный класс `prediction`,
- вероятность положительного класса `probability`.

Данные накапливаются в памяти сервиса. Для этого реализован отдельный класс `DriftMonitor`, который хранит события в буфере.

При старте сервиса запускается фоновая корутина, которая периодически проверяет, накопилось ли достаточное число событий для анализа. Если накоплен первый батч, он сохраняется как `reference_data`. Следующий батч используется как `current_data`.

Для сравнения `reference_data` и `current_data` используется отчёт Evidently:
`Report(metrics=[DataDriftPreset()])`.

В отчёт включаются как входные признаки модели, так и её выходы (`prediction` и `probability`), что позволяет отслеживать:
- дрифт входных данных,
- дрифт распределения предсказаний,
- дрифт распределения вероятностей модели.

Сформированный отчёт отправляется в Evidently Workspace по адресу, заданному в `EVIDENTLY_URL`, в проект с идентификатором `EVIDENTLY_PROJECT_ID`.
